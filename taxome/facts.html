GERARDO LAMAS INFO 5 May 2002

I believe that within 3-5 years I could compile a complete synonymical list
for the butterflies of the world, in database format, including not only a
list of all available names, but ALL names ever applied to butterflies.
This is already done for the Neotropical butterflies (>7579 "valid" species
at the latest count, Lamas et al. in press). and partly done for the Australian and Afrotropical
butterflies (in a less comprehensive format).  There is a skeleton list for
European species, and Harish Gaonkar is working on a list of the Indian
species.  Greatest needs are to compile exhaustive lists for the Holarctic
(kind of a nightmare because of all the synonyms, forms, aberrations, etc.
involved) and SE Asia.  The list of butterfly & moth genera compiled by the
NHM is of tremendous help, but needs to be checked carefully (I know quite a
number of errors and omissions in it).  Bridges' compilations are also
helpful, but he missed too many things and commited too many errors, as he
did not have access to all the primary literature.  Lastly, the NHM card
catalogue is pretty good, but is certainly not complete (even through
1980!).  With enough funds and paid personnel, it should be relatively easy
to incorporate into the core nomenclatural database digital images of the
primary types held in european collections (where more than 3/4 of all
extant types should be housed), and pdf's of the original descriptions.

GERARDO LAMAS INFO RE: GBIF PREPROPOSAL
Here in my office I'm sitting on three main sets of data which ought to go
into the public domain some day:
1. A virtually complete (from 1628 to the end of 2001) bibliography of the
Neotropical butterflies, all currently in Word (not databased), backed by
originals or photocopies of 99.9% of all references.
2. A complete checklist of the Neotropical butterflies, fully synonymical,
containing all genus group and species group names.  The list is in
WordPerfect (easily translatable into Word), but not databased.  I'm NOT
sole author of this list, as others involved are Mielke (Hesperiidae),
Robbins (Eumaeini), Casagrande (Brassolini), Viloria & Pyrcz (coauthors of
Pronophilina), and Callaghan (first author of Riodinidae).  Naturally, this
list evolves daily, as new information or interpretation (or speculation)
pours in.
3. A collection of photographs (about 95% prints, 5% slides) of primary
types (holotypes, lectotypes, neotypes, or at least one syntype) of ca. 75%
of the names ever applied to Neotropical butterflies (this possibly
represents over 90% of all curently existing types, the remainder are most
probably lost).  Quality of the photographs is quite variable, as they were
not taken and developed under the same conditions, but all are potentially
digitisable (and digitally enhanceable).

Things I might do at the NHM (as a sort of "consultant"), provided enough
time and resources were available:

1. Search for and locate "missing" Neotropical types (my rough estimate is
that some 200 Neotropical unrecognized types might still be found hidden in
odd drawers). Set those types (and others?) aside for simultaneous/ulterior
digitisation (by somebody else).
2. Do the same for all non-Neotropical types of Riodinidae.
3. Work towards completion of fully synonymical world checklists (in
database format) of some groups; Riodinidae and Hesperiidae would probably
be the easiest and fastest to complete, as they involve a relatively small
number of names in the Holarctic and Oriental literature.
4. Check the Viadocs database and (for Neotropical taxa) incorporate missing
names, correct all spelling and dating errors and, as far as feasible,
incorporate data on types (status, number, location, type-locality, etc.).

Regarding your questions about producing checklists or taxonomic lists, I don't see any problem there.  As long as the database is suitably structured (and George's is fine) it ought to be possible to output the data in any desired format or combination.  Similarly, there should be no problem in selecting only a subset of the data for output.

DSML is a relatively limited (and therefore limiting) database management system.  As you say, it was developed at NHM, but its originator subsequently left us, and although the concept is excellent, development has been patchy.  Chris Sleep (the man now responsible for mantaining and developing it) has managed to develop it to some extent, but is hampered in this regard by his many other responsibilities, so it is unlikely that DSML will ever become much more that what it currently is (although it is more than adaquete for producing formatted lists, and as you can see on the NHM site, it is possible to produce classy output - but it could be easier!).  There is so little documentation for DSML, and it's so hard to understand, that I wouldn't recommend it!

That said, any of the other DBMS packages that are available (asp, jsp, cold fusion, php) could be used to do the job, and in fact might do it better (or easier).

Would such systems be interoperable with others' querying systems?

I'm not uqite sure what you mean by the above.  It's the database that gets queried, and since almost all databases these days understand SQL, the database could be queried by any number of systems.  DSML, asp, jsp, or whatever, simply acquire and format the data, usually via ODBC or similar.

If you are talking about www applications, it really doesn't matter too much which you use.  The point is that the output must be comprehensible by the user's browser.  Since all server-side systems basically spit out HTML, more-or-less any user can access the output over the web (for a list or other simple output - the LepIndex site doesn't work well with version 3 browsers because of the javascript involved, but I wouldn't recommend anyone to use so old a browser unless they had no choice).  Similarly, if the database became much larger, or it so many users were accessing it that performance was adversly affected, it could be transferred into SQLserver (or a similar dedicated db system) and still be queried by the same www scripts, probably with only minor modifications.

A QUICK GUIDE TO DATABASE MANAGEMENT

DBMS -- DataBase Management Systems: all of them, online and not online, I guess.  These are programs which obtain information from databases, almost always these days via a standardized language, SQL.

ODBC -- Open Sources DataBase Connectivity, also called "Data Sources" in Win2000+, a software-independent method of finding information in databases. ODBC knows locations, etc. of databases, and versions of SQL used by each database program.  ODBC provides the infrastructure that allows SQL queries between databases constructed by different programs (e.g. Paradox, Access etc.).

SQL -- Structured Query Language, standardized language used to probe or query databases for information.

ASP  -- Active Server Pages, an open source DBMS for WWW, like all those below, spits out HTML "on the fly".  Microsoft's version is called ASP.NET, and they claim it is easier, more intuitive to write, etc.  All "scripts" code is visible, it is an interpreted language, unlike "Components", which are compiled I guess.

JSP -- JavaServer Pages.  Java is a programming script language ?designed with the web in mind.  JSP uses Java Script to manage data queries in its DBMS function.

coldfusion -- from MacroMedia: "Easily build Rich Internet Applications that integrate with databases, XML, web services, Macromedia Flash, and more."  Another DBMS.

PHP -- A DBMS which " is a widely-used general-purpose scripting language that is especially suited for Web development and can be embedded into HTML."  Initials stand for "PHP: Hypertext Preprocessor" (the inclusion of the initials in the title is a typical open source programmer's joke).  It is the most popular of all Apache WebServer add-ons.  Apache is also an open source program.  

SQLServer -- DB program for the web which should allow faster download of information.
